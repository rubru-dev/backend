# Prompt: k6 Load Test — RubahRumah (50 VU Concurrent)

## Konteks
E2E test suite sudah 133/133 pass dengan response time jauh di bawah threshold (sequential).
Sekarang kita test **concurrent load** — simulasi banyak user akses bersamaan — untuk memastikan
tidak ada race condition, connection pool exhaustion, atau degradasi performa saat multi-user.

Backend: Node.js + Express + Prisma + PostgreSQL
Data: 10.000+ leads, 5.000 stock opname logs, 1.000 laporan harian (sudah ada dari seed sebelumnya)

## Langkah 4 — `shared/auth.js`

```javascript
import http from "k6/http";
import { BASE_URL } from "./config.js";

/**
 * Login dan return token.
 * Dipanggil di setup() setiap script — token di-share ke semua VU via data passing.
 */
export function login(email, password) {
  const res = http.post(
    `${BASE_URL}/auth/login`,
    JSON.stringify({ email, password }),
    { headers: { "Content-Type": "application/json" } }
  );
  if (res.status !== 200) {
    throw new Error(`Login failed for ${email}: ${res.status} ${res.body}`);
  }
  return JSON.parse(res.body).access_token;
}

export function authHeaders(token) {
  return {
    headers: {
      Authorization: `Bearer ${token}`,
      "Content-Type": "application/json",
    },
  };
}
```

### `load-leads.js` — 50 VU Concurrent

**Skenario:** Simulasi 50 BD/Sales staff akses leads bersamaan dengan berbagai filter.

```
Stages:
  0→50 VU dalam 30 detik  (ramp up)
  50 VU selama 2 menit    (steady load)
  50→0 VU dalam 15 detik  (ramp down)
Total: ~2 menit 45 detik
```

**Thresholds:**
```
http_req_duration p(95) < 500ms   ← leads harus tetap cepat
http_req_duration p(99) < 1000ms
http_req_failed   rate  < 0.01
```

**Skenario per VU (random tiap iterasi):**
- 40% chance: GET /bd/leads?page={random 1-50}&limit=20
- 20% chance: GET /bd/leads?status={Hot/Medium/Low}&page=1
- 20% chance: GET /bd/leads?bulan={1-12}&tahun=2025&page=1
- 10% chance: GET /bd/leads?search=SEED_LEAD_{random}&page=1
- 10% chance: GET /bd/leads?status=Hot&bulan=3&tahun=2026 (filter combined)

**Checks per request:**
- `res.status === 200`
- `res.json().data !== undefined`
- `res.json().total >= 10000`
- Duration < 1000ms

### `load-write.js` — 20 VU Concurrent (Write Operations)

**Skenario:** Simulasi user create/update data bersamaan. Ini test paling penting untuk race condition.

```
Stages:
  0→20 VU dalam 20 detik
  20 VU selama 60 detik
  20→0 VU dalam 10 detik
```

**Thresholds:**
```
http_req_duration p(95) < 1000ms
http_req_failed   rate  < 0.02   ← sedikit lebih longgar untuk write
```

**Skenario per VU (setiap iterasi buat + hapus data sendiri):**

1. POST /bd/leads → simpan ID
2. PATCH /bd/leads/:id status → Hot
3. POST /laporan-harian (modul: bd)
4. DELETE /laporan-harian/:id
5. DELETE /bd/leads/:id

**Checks per step:**
- Create → status 201
- Update → status 200, data berubah sesuai
- Delete → status 200
- Tidak ada 409 Conflict atau 500

**Hal kritis:** Pastikan tidak ada ID collision antar VU. Gunakan `__VU` (virtual user ID) + `__ITER` (iteration count) sebagai suffix nama untuk uniqueness:
```javascript
const nama = `LOAD_TEST_VU${__VU}_ITER${__ITER}`;
```

### `stress-test.js` — Ramp Up Cari Breaking Point

**Tujuan:** Bukan untuk pass threshold, tapi untuk tahu di berapa VU sistem mulai degradasi.

```
Stages:
  0→50 VU   dalam 30 detik
  50 VU     selama 30 detik
  50→100 VU dalam 30 detik
  100 VU    selama 30 detik
  100→150 VU dalam 30 detik
  150 VU    selama 30 detik
  150→200 VU dalam 30 detik
  200 VU    selama 1 menit
  200→0 VU  dalam 30 detik
Total: ~6 menit
```

**Thresholds (lebih longgar — kita hanya observe):**
```
http_req_duration p(95) < 5000ms
http_req_failed   rate  < 0.10   ← alert kalau >10% error
```

**Endpoint yang ditest:** GET /bd/leads (paling representative)

**Yang diperhatikan di hasil:**
- Di VU berapa p95 mulai naik drastis?
- Di VU berapa error rate mulai > 1%?
- Apakah ada memory leak? (response time terus naik meski VU stabil)

## Hal Kritis yang Harus Diperhatikan

### Connection Pool
Prisma default connection pool = `num_cpus * 2 + 1`. Dengan 50 VU concurrent, kemungkinan hit pool limit. Cek di `schema.prisma`:
```
datasource db {
  url = env("DATABASE_URL")
  // Tambahkan jika perlu:
  // connection_limit = 20
}
```
Atau via DATABASE_URL: `postgresql://...?connection_limit=20&pool_timeout=10`

### Race Condition di Write Test
`load-write.js` paling rawan. Pastikan:
- Nama lead pakai `VU${__VU}_ITER${__ITER}` — tidak ada collision
- Delete hanya ID milik VU itu sendiri
- Jika ada 409/500, log endpoint + body untuk debug

### Token Expiry
Access token expire 15 menit. Jika script berjalan > 15 menit:
- Di `setup()` ambil token
- Di `teardown()` tidak perlu refresh (script sudah selesai)
- Untuk script > 15 menit, implement refresh di dalam VU loop

### Sleep / Think Time
Tanpa sleep, k6 akan spam request secepat mungkin — tidak realistis.
Gunakan `sleep(Math.random() * 2 + 0.5)` antar request untuk simulate user behavior.
